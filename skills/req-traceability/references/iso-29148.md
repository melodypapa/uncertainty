# ISO/IEC/IEEE 29148:2018 Reference

## Standard Overview

ISO/IEC/IEEE 29148:2018 defines requirements engineering processes for systems and software.

## Requirement Sections

### Functional Requirements

**Definition:** What the system shall do.

**Characteristics:**
- Describes system behavior, features, and capabilities
- Specifies inputs, outputs, and transformations
- Includes error handling and exception behavior

**Writing guidelines:**
- Use "System shall" language
- Focus on observable behavior
- Include specific conditions and constraints

### Non-Functional Requirements

**Definition:** Quality attributes and constraints.

**Categories:**
- **Performance:** Response time, throughput, scalability
- **Security:** Authentication, authorization, encryption
- **Reliability:** Availability, fault tolerance, recovery
- **Usability:** Learnability, efficiency, satisfaction
- **Maintainability:** Modularity, testability, documentation

**Writing guidelines:**
- Use measurable criteria (numbers, percentages)
- Reference industry standards where applicable
- Include specific thresholds and tolerances

### Interface Requirements

**Definition:** External system interactions.

**Types:**
- **User interfaces:** GUI, CLI, web UI
- **API interfaces:** REST, GraphQL, RPC
- **Hardware interfaces:** Sensors, actuators, devices
- **Database interfaces:** SQL, NoSQL, file systems

**Writing guidelines:**
- Specify protocol and data formats
- Define error handling for interface failures
- Include version compatibility requirements

### Data Requirements

**Definition:** Data structures and handling.

**Areas:**
- **Data models:** Entities, relationships, schemas
- **Data validation:** Rules, constraints, sanitization
- **Data storage:** Persistence, archiving, backup
- **Data privacy:** Encryption, anonymization, retention

**Writing guidelines:**
- Specify data types and formats
- Define validation rules and constraints
- Include data lifecycle requirements

### Verification Criteria

**Definition:** How to verify each requirement is met.

**Types:**
- **Test cases:** Unit tests, integration tests, end-to-end tests
- **Inspections:** Code reviews, walkthroughs, audits
- **Analysis:** Performance profiling, security scans
- **Demonstration:** User acceptance testing

**Writing guidelines:**
- Specify measurable acceptance criteria
- Include test scenarios and expected outcomes
- Reference relevant test suites or tools

## Requirement Attributes

| Attribute | Required | Format | Example |
|-----------|----------|--------|---------|
| ID | Yes | SWR_{CATEGORY}_##### | SWR_AUTH_00001 |
| Type | Yes | Functional/Non-Functional/Interface/Data | Functional |
| Text | Yes | "System shall..." | "System shall authenticate users" |
| Priority | Yes | Critical/High/Medium/Low | Critical |
| Status | Yes | Draft/Approved/Implemented/Rejected | Draft |
| Verification | Yes | Test/inspection method | "Verify login with valid credentials" |
| Parent_ID | No | SWR_{CATEGORY}_##### | SWR_AUTH_00001 |
| Source | No | File path or "manual" | src/auth/login.py |
| Rationale | No | Business justification | "Security requirement" |

## Writing Standards

### Language Rules

- Use **shall** for mandatory requirements
- Use **should** for recommended practices
- Use **may** for optional features
- Use **will** for factual statements about implementation

### Structure Rules

- One requirement per item
- No compound requirements (split into separate items)
- Include verification criteria for every requirement
- Use consistent ID format throughout

### Clarity Rules

- Avoid ambiguous terms (fast, good, adequate)
- Use specific, measurable criteria
- Include context and constraints
- Define technical terms

## Traceability

Requirements traceability matrix (RTM) links:
- Requirements to design documents
- Requirements to implementation code
- Requirements to test cases
- Requirements to user stories

Each requirement should maintain traceability through its lifecycle.
